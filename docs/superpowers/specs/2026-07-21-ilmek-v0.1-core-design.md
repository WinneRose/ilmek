# ilmek — v0.1 Core Design

**Status:** approved to build (user directive "start to write" / "go on", 2026-07-21)
**Source of truth:** the Notion project space *🧩 Turkish Lemmatizer & Stemmer* and its
subpages (*Yapay Zeka Yönergeleri*, *CLAUDE.md*, *AGENTS.md*). This document records the
decisions taken to begin implementation; where it and Notion disagree, Notion's product
intent wins and this file is corrected.

## 1. What we are building

An **explainable, deterministic, native Turkish morphology engine** exposing three views of
one analysis: `stem`, `lemmatize`, `analyze` (word level) and `analyze_sentence` (sentence
level). Not a suffix stripper — stem and lemma are *derived from the same morphological
analysis*, per the spec.

The wider product is a batteries-included Turkish NLP suite with optional Stanza/Zemberek
backends; **this pass delivers only the v0.1 native core** (roadmap M0→M2 partial). Later
milestones (Stanza adapter, full verb morphology, evaluation suite, disambiguation) are
out of scope here but the package layout reserves their homes.

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Package / import name | `ilmek` | Matches the repository the user created. |
| Public API | `stem`, `lemmatize`, `analyze`, `analyze_sentence` + `AnalysisResult` | Verbatim from spec; backward-compat contract. |
| Language / packaging | Python ≥3.10, hatchling, `pyproject.toml` | Modern, standard, light. |
| Morphology engine | Custom pure-Python **analysis-by-generation** (morphotactic FSM + morphophonemic realizer) | Deterministic, explainable, zero heavy deps, no network — matches "native backend works offline". FST library remains a future option behind the `Analyzer` interface. |
| Rules location | Data files (`ilmek/data/`) — lexicon JSON, morphotactic graph, phonology tables | "Rules in data files, extensible without code changes" (design principle). |
| License | Apache-2.0 | Per spec. |
| Tests | pytest, structured positive / negative / exception / consistency | Testing Contract in AGENTS.md. |

## 3. Architecture (layers, no cycles)

```
text
 → core.normalization      Turkish I/İ/ı/i, Unicode NFC, apostrophe standardization
 → core.tokenization       word / number / date / abbreviation / punctuation split → Token
 → morphology.lexicon      root lookup (trie over allomorphs), root attributes
 → morphology.morphotactics  noun & verb suffix-transition graph (FSM, data-driven)
 → morphology.phonology    vowel harmony (A, I), voicing, devoicing/assimilation, buffers
 → morphology.analyzer     analysis-by-generation → candidate AnalysisResult list
 → morphology.stemmer / lemmatizer   views over the best analysis
 → (future) disambiguator  sentence-context candidate ranking — SEPARATE, optional layer
```

**Boundary rule (from AGENTS.md):** the core analyzer is context-free and deterministic; the
disambiguator is an optional layer, never a required dependency of the core. No circular
dependencies between layers.

### Engine: analysis-by-generation

Reverse suffix-stripping with reversed phonology is fragile (one surface → many phonological
pre-images). Instead we **generate forward and match**:

1. Trie of root allomorphs (`kitap`, and auto-derived voiced `kitab`) yields every lexicon
   root that is a prefix of the surface word.
2. From each root, DFS the morphotactic FSM. Each transition carries an *archiphonemic*
   suffix (e.g. `lAr`, `DAn`, `(y)AcAk`); the phonology realizer turns it into its surface
   form given the running left context (last vowel → harmony, last phoneme → voicing).
3. Prune any path whose generated prefix diverges from the surface word; **accept** a path
   that exactly equals the surface word at a final state.
4. Every accepted path *is* an explanation: ordered morphemes + features + source=`lexicon`.
5. If no lexicon root produces the word, emit `source="guess"` candidates (unknown-root
   backoff) — kept strictly separate from lexicon-verified analyses.

This satisfies: explainable, deterministic, multiple candidates, morphophonemically valid
by construction, and the `normalization → … → morphophonemic validation → candidates` order.

### `AnalysisResult` (the shared contract)

```
AnalysisResult(surface, lemma, stem, pos, morphemes, features,
               analyses=[], confidence=None, backend="native", source="lexicon")
```
`confidence` is `None` unless a scoring layer sets it (no fabricated scores). `analyses`
holds alternative candidates. `to_dict()` / JSON export provided; CoNLL-U export is a later
milestone stub.

## 4. v0.1 morphological coverage

- **Nouns:** plural `lAr`; possessive 1sg/2sg/3sg/1pl/2pl/3pl; case acc/dat/loc/abl/gen/ins;
  pronominal `-n-` buffer after 3rd possessive; `(y)`/`(s)` buffers; genitive `(n)In`.
- **Verbs (basic):** negation `mA`; tense/aspect/mood progressive `Iyor`, future `(y)AcAk`,
  aorist, past `DI`, evidential `mIş`; copular stacking (ek-fiil) `mIş`/`DI`; person sets
  type-1 and type-2. Enough to analyze `gelmeyecekmişsiniz → gel(+neg+fut+evid+2pl)`.
- **Phonology:** 2-way (`A`) and 4-way (`I`) vowel harmony; stem-final voicing p/ç/t/k →
  b/c/d/ğ (lexicon `voicing` attribute); suffix `D` devoicing after voiceless; buffer
  y/s/n; vowel-drop roots (`burun→burnu`) via lexicon `forms`; `-Iyor` stem narrowing;
  irregular `de-`/`ye-` via lexicon `forms`.

Out of scope for v0.1: derivational morphology, full aspect/mood inventory, clitics `-ki`,
statistical disambiguation, Stanza/Zemberek backends (interfaces reserved only).

## 5. Data formats

Lexicon entry:
```json
{"lemma": "kitap", "pos": "NOUN", "attributes": ["voicing"], "forms": ["kitap", "kitab"]}
```
`forms` is optional; when `voicing` is present and absent, the voiced allomorph is derived.
Morphotactics and person/case tables live as Python-declared data (`morphotactics.py`) so
they are one import away yet still "rules as data" and trivially externalizable to JSON.

## 6. Testing & Definition of Done (this pass)

- `pip install -e .` succeeds; `import ilmek` exposes the four API functions.
- Canonical examples resolve: `kitaplarımızdan→kitap`, `evlerimizden→ev`,
  `gelmeyecekmişsiniz→gel`, plus voicing (`kitabı→kitap`) and harmony cases.
- Every phonology/morphotactic rule has positive + negative (+ exception where relevant)
  tests; a stem/lemma/analyze **consistency** test; pytest suite green.
- `AGENTS.md`, `CLAUDE.md`, `README.md`, `LICENSE`, CLI present; commands filled in.

## 7. Known limitations (carried forward, not hidden)

Small seed lexicon (the biggest lever for OOV coverage); verb inventory partial; no
disambiguation (multiple candidates returned unranked, `confidence=None`). The unknown-word
guesser is intentionally **conservative**: it strips a root only on strong evidence (a
multi-suffix chain or a single suffix ≥3 chars) and otherwise returns the surface as its own
stem, so it never mis-strips an ambiguous ending (`kalem` stays `kalem`, not `kale`). It
reuses the morphotactic FSM, so it strengthens automatically as morphology grows.

**OOV roadmap sequencing** (chosen 2026-07-21): (1) smarter guesser — *done here*;
(2) irregular pronouns / closed classes (`sen`, `var`, demonstratives) — small, near-term;
(3) derivational morphology (`-DIk`, `-mA`, `-lI`, `-sIz`, `-CI`…) at M3, which lets the
guesser and the analyzer reach roots like `yaşadıklarımızın → yaşa`. Expanding the lexicon
(the 200 staged entries + an open Turkish root list) remains the highest-impact complement.
